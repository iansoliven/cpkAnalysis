Manual Verification Checklist

This checklist complements the automated tests and is designed for QA engineers or stakeholders performing hands-on validation of the CPK Analysis solution.

Preparation

  • Install prerequisites as described in Getting Started.
  • Ensure sample STDF files are available (for example, the examples/ directory).
  • Close any open instances of the target workbook before running post-processing actions.

Core Workflow

  1. Run the pipeline: python -m cpkanalysis.cli run examples/lot2.stdf --output temp/ManualCheck.xlsx --postprocess.
  2. When the post-processing menu appears, exercise each action in “all tests” mode and verify workbook cells and charts update as described.
  3. Repeat in “single test” mode with target CPK values to confirm calculations affect only the selected test.
  4. Inspect the metadata JSON (temp/ManualCheck.json) and confirm the post_processing.runs array records each action with scope, tests, and parameters.

Console GUI

  1. Launch python -m cpkanalysis.gui and complete a run using interactive prompts.
  2. Accept the prompt to open post-processing immediately, then use the post command to reopen the menu.
  3. Verify that post_processing_profile.toml reflects the selected plugin configuration and persists between sessions.

Workbook Review

  • Open the generated workbook in Excel.
  • Confirm that the template sheet reflects updated ATE, Spec/What-If, and Proposed columns as appropriate.
  • Check charts (Histogram, CDF, Time-Series) for new limit markers and CPK text positioned above the title.
  • Review the hidden _PlotAxisRanges sheet for updated axis metadata.

Error Scenarios

  • Intentionally remove a required sheet (for example, “Test List and Limits”) and confirm the menu reports a descriptive error.
  • Open the workbook in Excel to lock the file, then attempt to save from the menu; ensure a warning is displayed.
  • Modify the workbook externally between menu sessions and use the Reload option to pick up changes.

Reporting

  • Capture workbook paths, timestamps, and screenshots of updated charts for review.
  • Record console logs (CLI or GUI) highlighting warnings or errors encountered during testing.
  • Summarise findings alongside automated test results when closing a test cycle.
