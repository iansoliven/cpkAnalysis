Manual Verification Checklist

QA procedures for hands-on validation and acceptance testing

Home Getting Started CLI Reference GUI Guide Testing Guide

This comprehensive checklist provides structured procedures for QA engineers, test engineers, and stakeholders performing hands-on validation of the CPK Analysis toolkit. These manual verification steps complement automated tests and ensure the solution meets quality standards in real-world scenarios.

Purpose of Manual Testing While automated tests validate code correctness, manual verification ensures usability, integration with Excel, chart quality, and end-to-end workflows meet user expectations. This checklist is especially valuable for:
  • Release qualification before deployment
  • User acceptance testing (UAT)
  • Training new team members on expected behavior
  • Documenting QA activities for regulatory compliance

Prerequisites and Preparation

Environment Setup

  • Required Complete installation per Getting Started guide
  • Required Python 3.11+ with all dependencies from requirements.txt
  • Required Sample STDF files available (e.g., examples/ directory)
  • Required Microsoft Excel or compatible spreadsheet software
  • Optional Test data with known characteristics for validation

Pre-Test Verification

Check Python Version

python --version  # Should be 3.11 or higher

Verify Module Import

python -c "import cpkanalysis; print('OK')"

Check CLI Help

python -m cpkanalysis.cli --help  # Should display command list

Verify Sample Files Exist

ls examples/*.stdf  # Should list STDF test files
Important: Clean State Before starting each test phase:
  • Close all Excel workbooks to avoid file locks
  • Clear temp/ directory of previous test outputs
  • Ensure sufficient disk space (>1GB for safety)

Phase 1: Core Pipeline Testing

Test 1.1: Basic Analysis Run

Objective: Verify end-to-end pipeline execution with default settings

Steps:

  1. Run pipeline with sample STDF file: 
    python -m cpkanalysis.cli run examples/lot2.stdf \
  --output temp/ManualTest_Basic.xlsx
  2. Observe console output for errors or warnings
  3. Verify completion message with output path
  4. Check files created:
    • temp/ManualTest_Basic.xlsx (workbook)
    • temp/ManualTest_Basic.json (metadata)

Acceptance Criteria:

  • Pipeline completes without errors
  • Console shows progress for ingestion, statistics, charts
  • Workbook file size > 100KB (indicates data present)
  • Metadata JSON contains test catalog and statistics

Test 1.2: Multiple File Processing

Objective: Verify batch processing of multiple STDF files

Steps:

  1. Run with multiple input files: 
    python -m cpkanalysis.cli run examples/lot2.stdf examples/lot3.stdf \
  --output temp/ManualTest_Multi.xlsx
  2. Verify both files are processed (check console)
  3. Open workbook and check device counts on CPK Report sheet

Acceptance Criteria:

  • Both files ingested successfully
  • Device count equals sum from both files
  • Test catalog includes all unique tests from both files

Test 1.3: Outlier Filtering

Objective: Verify outlier filtering affects statistics correctly

Steps:

  1. Run with IQR filtering: 
    python -m cpkanalysis.cli run examples/lot2.stdf \
  --outlier-method iqr --outlier-k 1.5 \
  --output temp/ManualTest_IQR.xlsx
  2. Run without filtering: 
    python -m cpkanalysis.cli run examples/lot2.stdf \
  --output temp/ManualTest_NoFilter.xlsx
  3. Open both workbooks side-by-side
  4. Compare mean, std dev, CPK values for same tests

Acceptance Criteria:

  • Filtered results show lower std dev (outliers removed)
  • CPK values typically higher with filtering
  • Metadata JSON documents filtering method and parameters

Test 1.4: Chart Generation Control

Objective: Verify chart disabling flags work correctly

Steps:

  1. Run with all charts disabled: 
    python -m cpkanalysis.cli run examples/lot2.stdf \
  --no-histogram --no-cdf --no-time-series \
  --output temp/ManualTest_NoCharts.xlsx
  2. Open workbook and check for chart sheets
  3. Run with only histograms: 
    python -m cpkanalysis.cli run examples/lot2.stdf \
  --no-cdf --no-time-series \
  --output temp/ManualTest_HistOnly.xlsx
  4. Verify only Histogram sheet exists

Acceptance Criteria:

  • NoCharts workbook has no Histogram/CDF/Time-Series sheets
  • HistOnly workbook has only Histogram sheet
  • CPK Report sheet present in both (statistics unaffected)
  • Processing time reduced when charts disabled

Phase 2: Console GUI Testing

Test 2.1: Interactive Workflow

Objective: Verify GUI prompts and configuration flow

Steps:

  1. Launch GUI: python -m cpkanalysis.gui
  2. Follow prompts, providing:
    • STDF file: examples/lot2.stdf
    • Template: Press Enter (use default)
    • Outlier method: none
    • Charts: Yes to all
    • Plugins: no
    • Output: temp/ManualTest_GUI.xlsx
  3. Wait for pipeline completion
  4. Decline post-processing menu
  5. Verify output files created

Acceptance Criteria:

  • All prompts clear and intuitive
  • Configuration summary displayed before execution
  • Pipeline runs successfully
  • Output matches CLI equivalent

Test 2.2: Post-Processing Integration

Objective: Verify seamless transition to post-processing

Steps:

  1. Launch GUI and complete analysis run
  2. Accept prompt to open post-processing menu
  3. Type help to see available commands
  4. Type audit to view (empty) log
  5. Type quit to exit

Acceptance Criteria:

  • Post-processing menu opens automatically
  • Commands execute without errors
  • Can exit cleanly back to shell

Phase 3: Post-Processing Menu Testing

Test 3.1: Update STDF Limits (All Tests)

Objective: Verify bulk limit updates and chart regeneration

Steps:

  1. Run pipeline with post-processing: 
    python -m cpkanalysis.cli run examples/lot2.stdf \
  --output temp/ManualTest_PP.xlsx --postprocess
  2. When menu appears, select option 1 (Update STDF Limits)
  3. Choose scope: All tests
  4. Provide target CPK: 1.67
  5. Wait for processing completion
  6. Save when prompted
  7. Exit menu
  8. Open temp/ManualTest_PP.xlsx in Excel
  9. Review CPK Report sheet: verify CPK values near 1.67
  10. Check histogram charts: verify limit markers updated

Acceptance Criteria:

  • Limit columns (LL, UL) updated for all tests
  • CPK values recalculated and close to target 1.67
  • Charts regenerated with new limit markers
  • Metadata JSON logs operation with timestamp and parameters

Test 3.2: Single Test Selection

Objective: Verify test search and single-test operations

Steps:

  1. Open post-processing menu for existing workbook
  2. Select option 1 (Update STDF Limits)
  3. Choose scope: Single test
  4. Search for test by typing partial name (e.g., "VDD")
  5. Select test from filtered list
  6. Provide target CPK: 2.0
  7. Save and exit
  8. Open workbook, verify only selected test changed

Acceptance Criteria:

  • Test search filters list correctly
  • Only selected test limits updated
  • Other tests remain unchanged
  • Charts for selected test regenerated

Test 3.3: Apply Spec Limits

Objective: Verify layered limit functionality

Steps:

  1. Open post-processing menu
  2. Select option 2 (Apply Spec / What-If Limits)
  3. Choose scope: All tests
  4. Provide target CPK: 1.33
  5. Save and exit
  6. Open workbook, check CPK Report sheet
  7. Verify LL_SPEC, UL_SPEC, CPK_SPEC columns populated
  8. Open histogram chart, verify both STDF and Spec limit markers visible

Acceptance Criteria:

  • Spec columns populated with calculated limits
  • STDF columns unchanged (non-destructive)
  • Charts show both limit types with different line styles
  • CPK text shows multiple values (CPK, CPK_SPEC)

Test 3.4: Calculate Proposed Limits

Objective: Verify proposed limit calculation and columns

Steps:

  1. Open post-processing menu
  2. Select option 3 (Calculate Proposed Limits)
  3. Provide target CPK: 1.5
  4. Choose scope: All tests
  5. Save and exit
  6. Open workbook, verify LL_PROP, UL_PROP, CPK_PROP, %YLD LOSS_PROP columns
  7. Verify CPK_PROP values equal or close to 1.5
  8. Check charts for "Proposed" markers

Acceptance Criteria:

  • All proposed columns populated
  • CPK_PROP matches target (1.5)
  • Yield loss calculated (typically very low for CPK 1.5)
  • Charts include proposed limit markers

Test 3.5: Audit Log Verification

Objective: Verify complete traceability of operations

Steps:

  1. Perform several post-processing operations (mix of single/all tests)
  2. In menu, select option 6 (View Audit Log)
  3. Review console output for all operations
  4. Open metadata JSON file, locate post_processing.runs
  5. Verify all operations logged with timestamps and parameters

Acceptance Criteria:

  • Every operation appears in audit log
  • Timestamps sequential and accurate
  • Parameters (scope, tests, target CPK) recorded
  • Warnings captured if any occurred

Test 3.6: Reload Functionality

Objective: Verify reload discards changes correctly

Steps:

  1. Open post-processing menu
  2. Make significant change (e.g., update all limits to CPK 2.0)
  3. Do NOT save
  4. Select option 5 (Reload Workbook)
  5. Confirm reload when prompted
  6. Make new change and save
  7. Open workbook, verify first change was abandoned, second change present

Acceptance Criteria:

  • Unsaved changes discarded
  • Workbook restored to disk state
  • Subsequent changes work normally
  • No audit entries for discarded operations

Phase 4: Workbook Review

Excel Workbook Inspection

Open a completed analysis workbook and verify the following:

CPK Report Sheet

  • All columns populated (TEST #, NAME, MEAN, STD DEV, LL, UL, CPK, etc.)
  • No #DIV/0 or #VALUE errors in cells
  • CPK values reasonable (typically 0.5 - 3.0 range)
  • Yield loss percentages make sense (0-100%)

Test List and Limits Sheet

  • Complete test catalog with test numbers and names
  • STDF limits populated where available
  • Spec/Proposed columns populated if post-processing applied
  • Units displayed correctly

Chart Sheets

  • Histogram: Bell curves visible, bins appropriate, limit markers present
  • CDF: Smooth S-curves, 0-100% range, limit markers at correct positions
  • Time-Series: Points vs. device index, trend visible if present
  • All charts have titles with test number and name
  • CPK values displayed above title
  • Legends clear and not overlapping data
  • Axes labeled with units

Hidden Metadata Sheet

  • _PlotAxisRanges sheet exists (may be hidden)
  • Contains axis ranges for all chart types
  • Data structured correctly for automation

Phase 5: Error Scenario Testing

Test 5.1: Missing Required Sheet

Objective: Verify graceful error handling for corrupted workbooks

Steps:

  1. Generate workbook normally
  2. Open in Excel, delete "Test List and Limits" sheet
  3. Save and close
  4. Attempt post-processing on modified workbook
  5. Observe error message

Acceptance Criteria:

  • Clear error message identifying missing sheet
  • No crash or stack trace
  • Menu exits gracefully

Test 5.2: File Lock Conflict

Objective: Verify handling of locked workbook files

Steps:

  1. Generate workbook
  2. Open workbook in Excel (leave open)
  3. Attempt post-processing save
  4. Observe warning/error
  5. Close Excel
  6. Retry save

Acceptance Criteria:

  • Warning displayed about file lock
  • User prompted to close Excel and retry
  • Save succeeds after Excel closed

Test 5.3: Invalid STDF File

Objective: Verify handling of corrupted or invalid input

Steps:

  1. Create dummy file: echo "invalid" > temp/bad.stdf
  2. Attempt analysis: 
    python -m cpkanalysis.cli run temp/bad.stdf \
  --output temp/Test_Bad.xlsx
  3. Observe error handling

Acceptance Criteria:

  • Error message indicates file format problem
  • No crash or incomplete workbook generated
  • Exit code non-zero

Reporting and Documentation

Test Report Template For each test cycle, document:
  • Environment: Python version, OS, Excel version
  • Test Date: When testing performed
  • Tester: Who performed verification
  • Results: Pass/Fail for each test phase
  • Issues Found: Description, severity, reproducibility
  • Screenshots: Capture key charts, error messages
  • Logs: Attach console output for failures
  • Files: Save generated workbooks and metadata

Acceptance Criteria Summary

For release approval, all of the following must pass:

  • Phase 1: All core pipeline tests pass without errors
  • Phase 2: GUI workflow completes successfully
  • Phase 3: All post-processing operations function correctly
  • Phase 4: Workbook structure and charts meet quality standards
  • Phase 5: Error scenarios handled gracefully with clear messages
  • No data corruption or loss observed
  • Metadata audit trails complete and accurate
  • Performance acceptable (typical STDF processes in < 2 minutes)

Tips for Effective Testing

Best Practices
  • Fresh Environment: Test on clean Python installation to catch missing dependencies
  • Different Platforms: Run subset on Windows, macOS, and Linux if possible
  • Real Data: Include tests with actual production STDF files (anonymized if needed)
  • Edge Cases: Try empty files, single-device files, thousands of tests
  • Concurrent Access: Verify behavior when multiple analyses run simultaneously
  • Long Sessions: Test post-processing menu with many iterations (memory leaks?)
Common Pitfalls to Watch For
  • Charts with overlapping legends or unreadable text
  • CPK calculations that don't match manual verification
  • Limit markers not visible on charts (wrong axis range)
  • Metadata JSON not updating after post-processing
  • Performance degradation with large STDF files (>100MB)
  • Memory usage growing unbounded during long sessions
